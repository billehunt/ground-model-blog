This week's lens: The State of AI Agents — Agents are moving from demos to production. We're examining what it actually takes to deploy reliable AI agents: the architecture decisions, the failure modes, and the builders doing it for real.

Lead Story: The Unsexy Truth About Production Agents — It's a Parsing Problem

AWS published a detailed walkthrough this week on building custom model providers for Strands Agents running on SageMaker endpoints. On the surface, it's a technical tutorial. Underneath, it's a confession about the actual state of AI agent deployment.

The problem: You deploy Llama 3.1 on SageMaker using SGLang (a popular serving framework). It returns OpenAI-compatible responses. You try to wire it into Strands Agents. You get TypeError: 'NoneType' object is not subscriptable. Your agent is dead on arrival — not because the model is bad, but because two perfectly functional systems speak different API dialects.

Strands expects Bedrock Messages API format. SGLang speaks OpenAI-compatible. Neither is wrong. Both are standard. And yet the gap between them requires you to write a custom LlamaModelProvider class that extends SageMakerAIModel to translate responses in real time.

Why this matters for builders:

  • The agent framework wars have a hidden cost. Every framework (Strands, LangChain, CrewAI, AutoGen) has its own expected response format. Every serving framework (SGLang, vLLM, TorchServe) has its own output format. The combinatorial explosion of parsers needed is the real integration tax nobody's pricing into their estimates.

  • "OpenAI-compatible" is a lie. It's more like "OpenAI-ish." Field names match, but structure, streaming behavior, tool-call formatting, and stop reasons all vary. If you're building production agents, budget 20-30% of your integration time for format translation alone.

  • This is why MCP matters. AWS is simultaneously pushing Model Context Protocol as a standardization layer. The pitch: one protocol to rule them all. The reality: it's another format to translate to and from, at least for now.

The builder's takeaway:

If you're deploying agents this quarter, don't start with the agent logic. Start with the plumbing. Map your model's actual output format against your framework's expected input. Write the parser first. Test it with streaming and non-streaming responses, with and without tool calls. The agent is only as reliable as the translation layer beneath it.

The unglamorous truth about the "age of agents": we're not blocked on intelligence. We're blocked on interoperability.

Company Watch: Strands Agents (AWS)

AWS's open-source Strands Agents SDK is quietly becoming a serious contender in the agent framework space:

  • Open source SDK with a model-driven approach — agents in a few lines of code
  • MCP integration for standardized tool and context access
  • SageMaker Unified Studio integration for end-to-end development
  • Reinforcement fine-tuning support for custom model behavior

AWS is playing the infrastructure game — not trying to own the agent logic, but making sure every path to production agents runs through their stack. For builders already in AWS, Strands is worth evaluating seriously. For those outside AWS, it's a signal that agent frameworks are becoming cloud-native platform features, not standalone tools.
